1 镜像及其获取策略

各工作节电负责运行Pod对象，而Pod的核心功用于运行容器，因此工作节电上必须配置容器运行引擎，如Docker等。启动容器时，容器引擎将首先于本地指定的景象文件，不存在的镜像则需要从指定镜像仓库(Registry)下载至本地：

Kubernetes 系统支持用户自定义镜像文件的获取策略，例如在网络资源为紧张时可以禁止从仓库中获取镜像文件等。容器 imagePullPolicy 字段用于为其指定镜像获取策略，它的可用值包括如下几个：

  • Always：镜像标签为"latest" 或镜像不存在时总是从指定仓库中获取镜像。
  • IfNotPresent： 仅当本地镜像缺少时方才从目标仓库下载镜像。
  • Never： 静止从仓库下载镜像，即使使用本地镜像。

2 暴漏端口

Docker 的网络模型中，使用默认网络的容器化应用需要过NAT机制将“暴漏” （expose）到外部网络中扩展中才能被其他节点之上的容器客户端所访问。然而，Kubernetes系统的网络模型中，各Pod的IP地址处于同一网络平面，无论是否为容器指定了要暴漏的端口，都不会影响集群其他节点之上的Pod客户端对其他进行访问，这就是意味着，任何监听在非lo接口接口上的端口都可以通过Pod网络直接被请求。从这个角度来说，容器端口只是信息性数据，它只是为集群用户提供一个快速了解相关Pod对象的可访问端口的途径，而且显式指定容器端口，还能为其赋予一个名称以方便调用。

  • containerPort ： 必选字段，指定在Pod对象的Ip地址上暴漏的容器端口
  • name ： 当前端口的名称，必须符合LANA_SVC_NAME规范且在当前Pod内必须唯一；此端口名可被Service资源调用
  • protocol： 端口相关的协议，其值仅可为TCP或UDP，默认为TCP

4. 环境变量

通过环境变量配置容器化应用时，需要在容器配置中嵌套使用env字段，它的值是一个由环境变量的 列表。环境变量通常由name和value字段构成。

  • name: 环境变量的名称
  • value: 传递给环境变量的值，通过$(VAR_NAME)引用，逃逸格式为$$(VAR_NAME)，默认值为空

5 共享节点的网络名称空间

同一个Pod对象的各容器运行于一个独立的，隔离的Network名称空间中，共享同一个网络协议栈及相关的网络设备。

6. 设置Pod对象的安全上下文

Pod对象的安全上下文用于设定Pod或容器的权限和访问控制功能，其支持设置的常用属性包括如下：

  • 基于用户ID（UID）和组ID（GID）控制访问对象（如文件）时的权限
  • 以特权或非特权的方式运行
  • 通过Linux Capabilites为其提供部分特权
  • 基于Seccomp过滤进程的系统调用
  • 基于SELinux的安全标签
  • 是否能够进行权限升级

Pod对象的安全上下文定义在spec.securityContext字段中，而容器的安全上下文则定义在sprc.containers[].securityContext字段中，且二者可嵌套使用字段还有所不同。
